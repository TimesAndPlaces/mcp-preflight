What a first scan tells you

Remote MCP URL uses insecure transport error

http://127.0.0.1:8080/mcp with inline credentials

Use HTTPS and move credentials out of the URL before first use.
Environment passthrough is too broad warning

inheritEnv=true and OPENAI_API_KEY forwarding

Reduce scope before the server gets host credentials it does not need.
Floating ephemeral launcher deserves review warning

npx @example/filesystem-server --path /

What it catches before trust

Not broad AppSec. Specifically MCP review.

This is the wedge: the setup-level mistakes and risky patterns generic tools often miss or explain poorly.

Remote MCP config risk

Insecure transport, credentials embedded in URLs, auth disabled, localhost and metadata targets.

Prompt and tool language

Prompt injection language, instruction bypass cues, tool poisoning, and suspicious side effects.

Environment and scope leakage

Broad filesystem paths, blanket environment inheritance, named token passthrough, and floating or opaque launchers.

Repo predictability

Missing lockfiles, unpinned specs, secret-bearing files, and install-source drift that makes review harder.

How a finding reads

Readable enough to act on.

What looked risky

Remote MCP server URL embeds credentials and uses plain HTTP.

Why it matters

URLs are easy to leak through logs and screenshots, and HTTP makes interception possible.

What to change next

Move credentials out of the URL, switch to HTTPS, and tighten the auth path before first use.
